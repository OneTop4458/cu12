# Session and Token Lifecycle

## Application Session (`cu12_session`)

1. Issued after successful authentication stage completion.
2. Stored as `httpOnly` cookie (`sameSite=lax`, `secure` in production).
3. TTL: 12 hours.
4. Invalid/expired token returns 401 for protected APIs.

## Login Challenge Token

1. Issued only when CU12 credentials are valid but user mapping does not exist.
2. Contains CU12 ID, campus, encrypted CU12 password, nonce, and purpose marker.
3. Short lifetime (5 minutes).
4. Must be exchanged with invite code in `/api/auth/login/invite`.

## Invite Token

1. Generated by admin API as plaintext once, then shown only at issuance time.
2. Stored as SHA-256 hash in DB (`tokenHash`).
3. Single-use, with expiration time.
4. Strictly bound to CU12 ID (`invite.cu12Id === challenge.cu12Id`).

## CU12 Credential Lifecycle

1. CU12 login is always re-validated at step-1 login.
2. Stored CU12 password is encrypted and updated on verified login.
3. Worker decrypts credential only during job execution.
4. Repeated login failures move account status to `NEEDS_REAUTH`.
